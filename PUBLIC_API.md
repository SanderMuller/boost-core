# Public API

The semver-protected surface of `sandermuller/boost-core`. Everything listed under **Stable surface** is covered by [Semantic Versioning](https://semver.org/spec/v2.0.0.html): it will not break in a MINOR or PATCH of the same MAJOR. Everything else вЂ” every class marked `@internal`, and all on-disk regenerable state вЂ” may change in any release, including patches.

## Versioning

This package follows Semantic Versioning 2.0.0. Pre-`1.0.0`, MINOR bumps may still break the public API (called out in `CHANGELOG.md` / `UPGRADING.md`). From `1.0.0` on, the surface below is locked for the `1.x` line.

## Stable surface

### Config authoring (`boost.php`)

- `SanderMuller\BoostCore\Config\BoostConfig` вЂ” the resolved config + `::configure()` entry point and its read accessors (`hasAgent`, `hasTag`, `isVendorAllowed`, `isEmitterDisabled`, `excludesSkill`, `excludesGuideline`), plus the public read properties (`agents`, `allowedVendors`, `tags`, вЂ¦). `BoostConfig::load(string $projectRoot, ?string $configFile = null): self` loads + builds a project's `boost.php` (the `@api` way to READ a declared config without the engine-internal loader; throws fail-loud on missing/invalid/ambiguous вЂ” the thrown classes `BoostConfigNotFoundException`, `InvalidBoostConfigException`, `AmbiguousBoostConfigException` are `@api`, catchable by name). The positional `__construct` is `@internal` (build via `configure()`).
- `SanderMuller\BoostCore\Config\BoostConfigBuilder` вЂ” the fluent builder: `withAgents`, `withAllowedVendors`, `withSkillsPath`, `withGuidelinesPath`, `withCommandsPath`, `withDisabledEmitters`, `withGitignoreManagement`, `withTags`, `withExcludedSkills`, `withExcludedGuidelines`, `withRemoteSkills`, `withSkillRenderers`, `withDisabledRenderers`, `withConventions`. (`build()` is `@internal` вЂ” the loader calls it.) The `withConventions([...])` array is consumer-authored slot DATA validated at sync against the active conventions-schema (the schema-version handshake under [Textual / wire formats](#textual--wire-formats)); its nested key vocabulary (`jira.*`, `pr.gates[]`, вЂ¦) is schema-defined by the vendor skills, not a literal frozen here.
- `SanderMuller\BoostCore\Enums\Agent` вЂ” the supported agent cases.
- `SanderMuller\BoostCore\Enums\Tag` вЂ” convenience tag cases (non-authoritative; any string is a valid tag). Existing case backing-values are stable; new cases may be added (additive вЂ” closes nothing). `withTags()` accepts `Tag|string`.
- `SanderMuller\BoostCore\Skills\Remote\RemoteSkillSource` + `RemoteSkillRef` вЂ” the `withRemoteSkills([...])` value objects (`::githubBundle()`, `::githubPath()`).

### Plugin contracts

- `SanderMuller\BoostCore\Contracts\FileEmitter` + `SanderMuller\BoostCore\Sync\SyncContext` (the received context) + `SanderMuller\BoostCore\Sync\EmittedFile` вЂ” emit custom files during sync. `emit()` returns `iterable<EmittedFile>` (zero = skip, one, or many). Parameterless constructors only. The context exposes `projectRoot`, the `BoostConfig`, and the installed-package set via `SanderMuller\BoostCore\Sync\InstalledPackages` / `PackageInfo` (also `@api`). `SyncContext` is received, not constructed by emitters.
- `SanderMuller\BoostCore\Contracts\SkillRenderer` + `SanderMuller\BoostCore\Skills\Rendering\RenderContext` (param), plus `PassthroughRenderer`, `InvalidSkillRendererException`, `SkillRenderException`.
- `SanderMuller\BoostCore\Contracts\BoostWrapperContract` + `SanderMuller\BoostCore\Agents\AgentTarget` вЂ” wrapper packages compute their emit surface against these. Only `AgentTarget`'s path/identity methods (`agent`, `skillsDirectoryRelative`, `guidelinesFileRelative`, `commandsDirectoryRelative`, `commandFileExtension`, `gitignorePatterns`) are `@api`; its `plan`/`format*`/`transpile*` methods are `@internal` (they operate on internal engine types).

### Wrapper integration (advanced)

For wrapper packages that drive a sync with INJECTED vendor skills/guidelines and EXTRA renderers (e.g. `project-boost-laravel` registering a Blade renderer and injecting `laravel/boost`'s assets). A plain consumer never needs this вЂ” the CLI + composer hooks cover the normal path.

- `SanderMuller\BoostCore\Sync\BoostSync` вЂ” the faГ§ade entry point. `BoostSync::make(?InstalledPackages, ?string $configFile): self`, then `->sync(string $projectRoot, bool $checkOnly = false, array $injectedVendorSkills = [], array $extraSkillRenderers = [], array $injectedVendorGuidelines = []): SyncResult`. The underlying `SyncEngine` stays `@internal` (free to evolve); this faГ§ade is the frozen entry point.
- `SanderMuller\BoostCore\Skills\Skill` + `SanderMuller\BoostCore\Skills\Guideline` вЂ” the value types injected via `injectedVendorSkills` / `injectedVendorGuidelines` (keyed by `vendor/package`). Their constructor properties are frozen; new ones append with a default (`Skill::$assets`, added 1.3, and `Skill::$requires` / `Skill::$requiresValid`, added 1.4, are such appends).
- `SanderMuller\BoostCore\Skills\SkillAsset` вЂ” the value type for a skill's companion files (`relativePath`, `contents`), passed via `Skill::$assets` to ship `scripts/` / `references/` siblings beside the emitted `SKILL.md`. Same frozen-constructor rule.
- `SanderMuller\BoostCore\Sync\SyncResult` вЂ” the return. Frozen read surface: `$writes` (list of `WrittenFile` вЂ” `relativePath`, `action`), `$emitters` (list of `EmitterResult` вЂ” `action`, `relativePath`, `fqcn`, `vendor`, `reason`), `$errors` (list<string>), `$diagnostics` (list of `Diagnostic` вЂ” `level`, `slot`, `message`, `vendor`), `$hostShadows` (list of `array{skill, shadowedVendor}`), plus `hasErrors()`, `countByAction(WriteAction)`, `countEmittersByAction(EmitterAction)`, `renderDeleteAttribution()`. The `WriteAction` / `EmitterAction` enums are `@api` (existing backing values stable; new cases additive). The `conventions*` properties are engine-internal, not frozen.
- **Named-argument contract.** Consumers pass named arguments to `@api` methods and instantiate `@api` value objects directly, so the PARAMETER NAMES are part of the frozen contract вЂ” a rename or reorder is a breaking change even when the class/method stays `@api`. Frozen: the constructor parameters of `Skill` / `Guideline` / `RenderContext` / `WrittenFile` / `EmitterResult` / `Diagnostic`, and the method parameters of `BoostSync::sync(projectRoot, checkOnly, injectedVendorSkills, extraSkillRenderers, injectedVendorGuidelines)` + `BoostConfig::load(projectRoot, configFile)`. Two value-shape pins the class-`@api` guarantee doesn't otherwise cover: `Diagnostic::$level` stays a STRING (`error`/`warning`/`info`) within `1.x` вЂ” not enum-ified without a major; and `SyncResult::$errors` stays `list<string>` (consumers interpolate entries directly).
- `SanderMuller\BoostCore\Agents\AgentTarget::skillRelativePathForName(string): string` вЂ” compute a skill's emit path (relative to `skillsDirectoryRelative()`) from its name, for a `BoostWrapperContract::injectedEmitPaths()` implementation вЂ” without constructing an engine-internal `Skill`. A wrapper injecting skills with `assets` must additionally claim each asset's emit path (`skillsDirectoryRelative()/<name>/<assetRelativePath>`) in `injectedEmitPaths()`, or a bare-CLI sync reaps the asset copies as stale.
- `SanderMuller\BoostCore\Enums\Agent::target(): AgentTarget` вЂ” the agent-value в†’ target bridge, so a wrapper resolves a `list<Agent>` to their `AgentTarget`s (and computes per-agent emit paths) without referencing the engine-internal concrete target classes.
- `SanderMuller\BoostCore\Skills\FrontmatterParser` + `SanderMuller\BoostCore\Skills\ParsedDocument` вЂ” `parse(string): ParsedDocument` (read `->frontmatter` / `->body`). Reuse it for boost-tag parity instead of rolling your own YAML-head parse. Pure, parameterless constructor.
- `SanderMuller\BoostCore\Skills\BoostTags` вЂ” the tag-parse seam, the other half of `FrontmatterParser`: `BoostTags::parse(array $frontmatter): array{0: list<string>, 1: bool}` (returns `[normalized tags, valid]`) and `BoostTags::declaresTags(array $frontmatter): bool`. A wrapper that injects skills/guidelines computes the SAME `[tags, valid]` the engine does вЂ” **including the fail-closed contract** (a `metadata.boost-tags` present-but-not-a-string в†’ `valid = false` в†’ ships nowhere) вЂ” instead of reinventing the tokenize+validate and diverging on the malformed case. Pure static, parameterless. (`parseString()` is the internal lexer вЂ” `@internal`, not part of the frozen surface.)
- `SanderMuller\BoostCore\Skills\BoostRequires` вЂ” the dependency-parse seam for `metadata.boost-requires` (added 1.4), mirroring `BoostTags`: `BoostRequires::parse(array $frontmatter): array{0: list<string>, 1: bool}` (returns `[required skill names, valid]`) and `BoostRequires::declaresRequires(array $frontmatter): bool`. Names are NOT case-folded (they compare exactly as skill names resolve), and вЂ” unlike the tag seam вЂ” a malformed value does NOT fail closed: the skill still ships, `valid = false` surfaces as a sync warning / `boost validate` error. A wrapper that injects skills computes the same pair via this seam and passes it as `Skill::$requires` / `Skill::$requiresValid`. Pure static, parameterless.

### Composer hooks

- `SanderMuller\BoostCore\Scripts\BoostAutoSync::run` / `runWithSummary` вЂ” the `post-install-cmd` / `post-update-cmd` targets.
- `BoostAutoSync::syncUserScope` / `syncUserScopeOnce` вЂ” in-process self-sync for globally-installed CLI tools.

New parameters on any stable method are always optional-with-default; their absence-vs-presence is not a breaking change.

### CLI (`bin/boost`)

The command names, their documented options, and the exit-code contract (`0` ok, `1` failure, `2` usage) are stable. Human-readable output text is NOT a contract; `--json` envelopes (where offered) are.

**Ambiguous config fails loud (behavior contract).** When BOTH a root `boost.php` and a `.config/boost.php` are present, config resolution fails loudly (non-zero exit) rather than silently picking one. This fail-on-ambiguous *behavior* is contractual, and the thrown `AmbiguousBoostConfigException` is `@api` вЂ” catchable by name (as are `BoostConfigNotFoundException` and `InvalidBoostConfigException` from `BoostConfig::load()`).

### Family-CLI extension point

- `SanderMuller\BoostCore\Commands\BoostBaseCommand` вЂ” the base class wrapper/tooling packages extend to ship their own `bin/<tool>` commands. The frozen `@api` surface is exactly two protected helpers: `addWorkingDirOption(): static` (registers the `--working-dir`/`-d` option) and `resolveProjectRoot(InputInterface): string` (resolves the project root, `--working-dir` в†’ cwd). Subclass via Symfony's normal `configure()` / `execute()`. The class's other protected members (the `--config` option + config-loading helpers) are `@internal` вЂ” a config-loading extension point is a separate, heavier contract deliberately not locked at 1.0.

### Textual / wire formats

These aren't PHP types, but they're authored or observed by consumers/publishers, so they're part of the contract (changing them is a major bump):

- **`boost:conv` inline token** вЂ” `<!--boost:conv path="вЂ¦" mode="вЂ¦" fallback="вЂ¦"-->` (and the ` ```boost:conv ` fence). Vendor skill/guideline authors hand-write these; the `\`-escape (`<!--\boost:convвЂ¦-->`) is also stable.
  - **Path syntax** (stable): the legacy `$.`-prefix, dotted `group.key` (`jira.project_key`), bare group (`mcp`), and open-vocab sub-key (`group.<sub-key>`, e.g. `mcp.<server>` вЂ” the `mcp.` prefix won't be repurposed).
  - **Modes + typeГ—mode validity** (frozen): the modes are `inline`, `bullets`, `yaml`, `json`, valid by slot value type as вЂ” scalar в†’ `inline`/`yaml`/`json`; list в†’ `inline`/`bullets`/`yaml`/`json`; map в†’ `yaml`/`json`; with `inline` additionally requiring a single-line scalar OR a scalar list (a multi-line scalar, or a list of structured items, rejects `inline` as a render-class error). (So a list-of-strings в†’ `inline` is stably valid.)
  - **Resolution behavior** (frozen): a token resolves `declared value в†’ schema default в†’ fallback`; an unset slot with a `fallback` inlines the fallback string verbatim (in the token's mode), and an unset slot with no default and no fallback is a render-class error that fails `boost sync --check`. The resolved value FULLY replaces the `<!--boost:convвЂ¦-->` comment вЂ” zero marker residue in emitted output (a surviving token is a leak, surfaced by `boost doctor` / `validate`).
  - **Paired visible-default form** (stable, added in 1.2.1): a token MAY wrap a visible default between an open and an end comment вЂ” `<!--boost:conv path="вЂ¦" mode="вЂ¦"-->VISIBLE<!--boost:conv:end-->` (the ` ```boost:conv ` fence takes the same `<!--boost:conv:end-->` marker and buffers its body as one block, so a multi-line span resolves whole). boost-core replaces the entire span (open comment в†’ end comment) with the resolved value; the VISIBLE text doubles as the inline fallback (an explicit `fallback=` still wins). Paired spans resolve BEFORE bare tokens so the open comment is never consumed as a stray unpaired token; an orphan `<!--boost:conv:end-->` is inert and keeps the Project Conventions block. The point is graceful degradation under a resolver-LESS engine (notably `laravel/boost`, which preserves the HTML comments verbatim): both comments stay inert and the VISIBLE default reads as ordinary prose вЂ” no word gap вЂ” where a bare token would leave a hole.
- **Conventions schema-version handshake** вЂ” the negotiation MECHANISM is frozen, not just its current instance. A vendor skill declares `metadata.schema-required` as a caret RANGE (e.g. `"^1"`); the emitted Project Conventions block carries `schema-version: N` as a const INTEGER; the engine ships the skill's slots only when the range вЉ‡ the const. Additive growth within a major (new optional slots) stays compatible under the same range; only a breaking redesign bumps both the integer and the required range. `^1` / `schema-version: 1` is the current (v1) instance. The host `schema-version: N` for an unpinned project is SEEDED as `max(minRequired across all allowlisted vendor schemas) ?? 1` (an operator may pin it explicitly). Because the host carries a SINGLE `schema-version`, all conventions-vendors allowlisted in one host must share a schema-version MAJOR: a co-allowlisted vendor requiring `^2` raises the shared host const and withholds every still-`^1` vendor's slots (with an error diagnostic). Multi-catalog contribution at DIFFERENT majors is therefore unsupported within `1.x` вЂ” per-vendor schema-version would be a future handshake redesign, not a `1.x` change.
- **Tag filtering вЂ” two co-equal channels, subset-AND** вЂ” a vendor skill/guideline ships to a project only when its declared tags вЉ† the project's `withTags()` set (an untagged skill/guideline ships everywhere; a malformed tag declaration fails CLOSED вЂ” ships nowhere). Tags are declared on two stable, **co-equal** channels: a SKILL via `metadata.boost-tags` in its frontmatter; a GUIDELINE via `metadata.boost-tags` frontmatter **or** the sidecar manifest `.boost-tags.yaml`. When a guideline carries frontmatter tags they win; otherwise the sidecar is read вЂ” but the sidecar is a **first-class, standalone, non-deprecated** channel (guidelines often must stay frontmatter-free for laravel/boost compatibility, so the sidecar is never relegated to second-class). Sidecar shape (frozen): a YAML map of exact guideline filename в†’ a space-delimited tag string (no list form, no globs). Tag comparison is case/space-normalized (`strtolower(trim(...))`) identically on the declared and the `withTags()` side.
- **Conventions exclude-key grammar** вЂ” the `vendor/package:name` form passed to `withExcludedSkills()` / `withExcludedGuidelines()`.
- **Managed `.gitignore` block markers** вЂ” `# >>> boost (managed) >>>` вЂ¦ `# <<< boost (managed) <<<` (written into the consumer's tracked `.gitignore`).
- **`Agent` enum backing values** вЂ” the kebab strings (`claude-code`, вЂ¦) double as on-disk directory slugs and cross the wrapper boundary; new agents may be added, existing values won't change.
- **Vendor publishing paths + override keys** вЂ” a vendor package ships skills/guidelines/schema at the default roots `resources/boost/skills/`, `resources/boost/guidelines/`, `resources/boost/conventions-schema.json`, overridable via the `extra.boost.skills` / `extra.boost.guidelines` composer.json keys. The default roots + the override-key names are frozen (a vendor that declares neither relies on the defaults; changing them silently un-discovers its catalog).
- **SKILL.md / guideline frontmatter keys boost-core CONSUMES** вЂ” exactly `name`, `description`, `metadata.boost-tags`, `metadata.schema-required`. boost-core reads + acts on these; their names + semantics are frozen (a vendor authors against them, so renaming/dropping one silently changes behavior). (The argument system uses the separate `arguments:` key + body placeholders вЂ” see the commands surface.)
- **Accepted skill-source shapes** вЂ” a skill source is discovered as EITHER a top-level flat file (`<name>.<ext>`) OR a depth-1 nested entry (`<name>/SKILL.<ext>`, with `references/` / `examples/` treated as assets, not skills). BOTH shapes are accepted and stay accepted within `1.x` вЂ” a future minor will not drop flat-file acceptance and silently un-discover a flat catalog. (The nested `<name>/SKILL.md` directory shape is the family-canonical authoring convention; flat remains equally valid.) Backup/editor-temp files (`*~`, `*.bak`/`.orig`/`.tmp`/`.swp`/`.swo`) are never treated as sources.
- **Asset-sibling emission (1.3)** вЂ” a nested skill's asset siblings (every non-`SKILL.*`, non-hidden, non-backup file under `<name>/`, at any depth) are emitted verbatim beside the rendered `SKILL.md` in every configured agent's skill directory, and are manifest-tracked like any other emitted file: removing an asset from the source (or removing/tag-filtering the whole skill) reaps the emitted copies on the next sync. Applies to host-authored, vendor, injected, and remote skills alike.
- **Frontmatter pass-through (behavior contract)** вЂ” a frontmatter key boost-core does NOT consume is preserved value-level through emission: the full frontmatter map round-trips into the destination `SKILL.md` (YAML re-dump may reorder keys; values are preserved). This protects Agent-Skills display fields (`argument-hint`, `license`, and future keys like `allowed-tools`/`model`) WITHOUT the engine enumerating them вЂ” and guarantees a future "emit only known keys" change can never silently drop them.
- **Conventions-schema `render` annotation** вЂ” the JSONSchema extension keyword `"render"` on a slot (a single mode string, or a list of allowed modes, drawn from `inline`/`bullets`/`yaml`/`json`) pins which token modes a slot permits вЂ” the schema-side complement to the token typeГ—mode matrix. The keyword + its value vocabulary are frozen.
- **Token forward-compat** вЂ” an unknown `boost:conv` `mode` (or a malformed token) is a render-class ERROR (surfaced + failing `sync --check`), NEVER a silent corruption or verbatim-leak of the token into output. This keeps the token format growable: a future mode is gated behind a boost-core floor bump, and an older engine fails loudly rather than mis-rendering.
- **Emitted-output layout** вЂ” the per-agent guidance files (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`, вЂ¦) and per-agent skill directories (`.{agent-slug}/skills/<name>/SKILL.md`, slug = the frozen `Agent` backing value) are the contracted emit locations.

## Internal (not covered by semver)

- Every class marked `@internal` вЂ” the engine: `Sync\`, `Discovery\`, `Conventions\`, `Agents\` (except `AgentTarget`), `Commands\`, the `Skills\` internals, `Env`, and the internal `Config\` loader/writer/printer/path classes. Do not import these.
- On-disk regenerable state: the sync manifest (`.boost/manifest.json` вЃ„ `.config/boost/manifest.json`), the remote-skill ledger (`remote-manifest.json`), the user-scope manifests under `~/.boost/manifests/`, the `.boost/` вЃ„ `.config/boost/` runtime dir, and the cache sentinel. Their schema is not a contract.

## Deprecation policy

A stable (`@api`) element is deprecated before it is removed: marked `@deprecated` in PHPDoc вЂ” and, where it has a runtime code path, emitting `E_USER_DEPRECATED` вЂ” in a MINOR release, then removed no earlier than the next MAJOR. Deprecations are listed under `### Deprecated` in `CHANGELOG.md` so they surface in release notes.
